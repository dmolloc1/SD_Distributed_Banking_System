ProblemaContexto del proyectoEn el panorama financiero contemporáneo, la interoperabilidad entre entidades bancarias es un requisito fundamental para la agilidad económica. El presente proyecto aborda la implementación de un Ecosistema Bancario Distribuido compuesto por tres entidades autónomas que operan en diferentes regiones geográficas, pero que funcionan bajo un modelo de federación técnica.El sistema permite que clientes con presencia en múltiples bancos realicen operaciones financieras: como transferencias, depósitos y retiros, de manera transparente, eliminando las fricciones de las fronteras bancarias tradicionales. La solución se fundamenta en el desarrollo de un core transaccional robusto que gestiona la consistencia de datos a través de una red de microservicios, utilizando el sistema de archivos como medio primordial de persistencia.Definición del problemaEl desafío técnico no reside únicamente en la capacidad de mover fondos entre cuentas, sino en garantizar la integridad financiera en un entorno donde no existe una base de datos centralizada. El enunciado propone una arquitectura donde cada banco es custodio exclusivo de sus datos, almacenados localmente en archivos, lo que introduce riesgos críticos de consistencia, concurrencia y atomicidad.Para ilustrar la complejidad del escenario propuesto, se presenta la siguiente comparativa entre el modelo bancario tradicional y el sistema distribuido desarrollado:CaracterísticaModelo Centralizado TradicionalSistema Bancario Distribuido (Propuesto)PersistenciaBase de Datos Relacional (ACID Nativo)Sistema de Archivos Local (JSON/LOG)Punto de FallaÚnico (SPOF en el servidor central)Distribuido (La caída de un banco no afecta a los otros)IdentidadRegistro único en base de datosIdentidad Federada (Clientes compartidos entre nodos)ConsistenciaTransacciones locales inmediatasTransacciones Distribuidas (Consenso entre nodos)EscalabilidadVertical (Hardware más potente)Horizontal (Inclusión de nuevos nodos independientes)Desafios técnicosEl desarrollo del sistema se ha enfrentado a cuatro pilares de complejidad que definen la naturaleza de los sistemas distribuidos:Identidad Federada: Resolver el mapeo de clientes que poseen cuentas en dos o tres bancos simultáneamente, garantizando una vista consolidada de su patrimonio sin duplicidad de datos.Exclusión Mutua: Gestionar el acceso concurrente a los archivos de cuentas. Dado que múltiples procesos pueden intentar modificar un saldo al mismo tiempo, el sistema debe implementar bloqueos que prevengan la corrupción de información.Atomicidad de Escritura: Garantizar que una operación en el sistema de archivos sea de tipo "todo o nada", evitando estados inconsistentes si el proceso se interrumpe durante una escritura en disco.Acuerdo y Coordinación: Implementar un protocolo de consenso para transferencias interbancarias, asegurando que un débito en el Banco A siempre se corresponda con un crédito en el Banco B, o que ambos se reviertan en caso de error.Objetivos del SistemaEl objetivo primordial es la construcción de un sistema financiero resiliente que cumpla con los estándares de un Distributed Ledger académico. Se diseñó un Mecanismo de Bloqueo Híbrido que opera en dos niveles:Bloqueo Interno (JVM): Utiliza un ReentrantLock para serializar el acceso de los hilos dentro del microservicio.Bloqueo Externo (OS): Utiliza FileChannel.lock() de Java NIO sobre un archivo de semáforo (.lock) para proteger el recurso contra otros procesos externos o contenedores.Esta combinación garantiza que el acceso a los saldos sea estrictamente secuencial, eliminando el riesgo de condiciones de carrera.Auditoría mediante Ledger ImmutableCada movimiento financiero se registra en un archivo transactions.log antes de impactar el saldo final. Este registro es inmutable (Append-only) y cuenta con un mecanismo de seguridad basado en Checksums SHA-256.A continuación se presentan los campos críticos del JournalCampoDescripciónTimestampMarca de tiempo con precisión de milisegundos.Transaction IDIdentificador único global de la operación.Previous/New BalancePermite validar la transición de estados.ChecksumFirma digital de la línea para detectar alteraciones manuales en el archivo de logs.Puerto y Adaptador de Persistencia (Port & Adapter)Para desacoplar la lógica de negocio de los detalles del sistema de archivos, se implementó el patrón Port & Adapter.
